# Knowledge Base Phase 2 Comparison

Date: 2026-06-26

Compared documents:

- `cardholder-phase1.md`
- `belt-phase2.md`
- `shoe-phase2.md`

Goal: prove whether one knowledge extraction structure can scale from simple leather goods to straps and footwear.

## Summary

The same knowledge-record structure works across all three products, but the required geometry model changes sharply:

- Cardholder: 2D planar pieces with pockets, stitch lines, holes, and print-ready scale.
- Belt: mostly 1D strap geometry plus hardware compatibility.
- Shoe: 3D-last-based product requiring surface flattening, fit validation, construction method, and prototype review.

The knowledge base scales if it stores not only formulas, but also confidence, source lineage, applicability, required inputs, derived outputs, and validation state.

## Universal Knowledge

These concepts appear in all three products:

- physical units must be explicit;
- material thickness affects correctness;
- print/export scale must be controlled;
- hardware/tool compatibility must be represented when present;
- cut lines, fold lines, stitch lines, holes, labels, and control marks are reusable pattern concepts;
- validation rules are as important as geometry;
- source-backed and master-validated rules must not be mixed;
- open sources rarely provide complete production tolerances;
- geometry should be deterministic and generated by code, not guessed by language models.

## Why This Scales

The model scales because it does not assume all products are the same shape. It stores knowledge as rules with applicability, confidence, and geometry impact.

Cardholder can be implemented from mostly 2D rules.

Belt can be implemented from mostly 1D strap rules plus hardware tables.

Shoe cannot be implemented safely from simple form inputs; the same structure still captures that reality by marking last geometry, flattening, and fit rules as required dependencies.

## Platform Implication

The future Pattern Engine should not be one engine with hard-coded products. It should be a rule-backed pipeline:

```txt
Product Type
-> Knowledge Rules
-> Input Schema
-> Validation
-> Derived Parameters
-> Geometry Model
-> Export Model
-> Prototype / QA State
```

This allows the platform to add simple products quickly while preventing complex products from being falsely presented as production-ready before the required expert knowledge exists.
